# Story 20.9: Secure One-Time Firmware Download Link

**Epic:** Epic 20 — Pluggable External Integrations & Firmware Lifecycle
**Phase:** PHASE 3: SECURITY
**Persona:** Michael (Sungrow Stakeholder) / Field Service Technician
**Priority:** P1
**Story Points:** 5
**Status:** New
**GitHub Issue:** #391
**Target URL:** `/deployment/firmware/:firmwareId/download`

## User Story

As an operations manager, I want to generate a secure, one-time-use download link for a firmware artifact, so that field service technicians can access the artifact exactly once with MFA verification and we maintain full audit trail.

## Preconditions

- Story 20.1 `IArtifactProvider.generateSecureLink()` method exists
- Story 20.6 firmware version detail page exists
- Cognito authentication with MFA is enabled (Epic 1)

## Context / Business Rules

- **Abdul's requirement:** "Give field service people a one-time link. They need MFA, we give them a code so they can download." This is a security-critical feature for NIST compliance.
- **One-time semantics:** Link is valid for a single download. After use or expiry (configurable, default 24h), the link is invalidated. Backend tracks link status.
- **MFA verification:** Before generating the link, the system requires the requesting user to complete MFA step-up. The link itself embeds a token that the field tech must verify with a code (sent via email/SMS).
- **Audit trail:** Every link generation, access attempt (success or failure), and expiry is logged as an audit event.
- **Provider-agnostic:** Uses `IArtifactProvider.generateSecureLink()` — works with JFrog, S3, or any adapter.

## Acceptance Criteria

- [ ] AC1: Firmware detail page (Story 20.6) has a "Generate Download Link" button visible only to authorized roles (Operations Manager, Admin)
- [ ] AC2: Clicking the button triggers MFA step-up verification (re-authentication or TOTP confirmation)
- [ ] AC3: After MFA, a modal shows the generated one-time link with: URL (copy button), expiry countdown, recipient field (email/phone for sending the verification code), and a "Send Link" action
- [ ] AC4: The generated link includes a unique token. Accessing the link shows a verification page requiring a 6-digit code (sent to the specified recipient)
- [ ] AC5: After correct code entry, the firmware artifact downloads and the link is invalidated (subsequent access shows "Link expired or already used")
- [ ] AC6: Link metadata (generated by, generated at, recipient, status: active/used/expired, used at, IP address) is stored and visible in the firmware's audit timeline
- [ ] AC7: Admin can view all active links and manually revoke them from a "Active Links" section on the firmware detail page
- [ ] AC8: `SecureLinkOptions` type supports: `expiresIn` (seconds), `maxUses` (default 1), `recipientEmail`, `recipientPhone`, `requireMFA` (default true)

## UI Behavior

- "Generate Download Link" button is a secondary action in the firmware detail header
- MFA step-up uses existing Cognito MFA flow (TOTP or SMS)
- Modal is a multi-step wizard: Step 1 — MFA verify, Step 2 — Configure link (recipient, expiry), Step 3 — Link generated (copy + send)
- Verification page is a standalone minimal page (no sidebar/nav — accessible to field techs who may not have full app access)
- Active links section shows a compact table with status badges

## Out of Scope

- SMS/email delivery infrastructure (uses existing notification provider or manual copy)
- Mobile app for field tech download (web-only for now)
- Bulk link generation (one at a time)
- Offline download support
